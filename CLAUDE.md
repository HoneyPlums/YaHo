# 잡아드림 (Job-a-Dream)

AI 기반 맞춤형 기업 추천 + 청년 지원정책 통합 플랫폼. 해커톤 MVP (개발 시간 5시간, 5인 팀, GitHub 단일 레포로 협업).
**대회 규정상 비용 지출이 불가능해, AI는 카드 등록 없는 무료 티어(Gemini API)만 사용한다.**
저장소: https://github.com/HoneyPlums/YaHo.git
배경/기능 우선순위/디자인 시스템/DB/AI 파이프라인의 전체 내용은 `PRD.md` 참고. 이 파일은 팀 전원이 Claude Code로 작업할 때 공통으로 지킬 최소 규칙만 담는다.

## 기술 스택 (2026-07-05 실제 구조 기준으로 갱신)
- Frontend: Vite + React, JavaScript, Tailwind CSS (모두 `FE/` 폴더 안). 원래 PRD/CLAUDE.md는 Next.js를 전제했지만, FE 담당이 이미 Vite로 구현을 시작해 **Vite 유지로 확정**했다
- Backend: Next.js Route Handlers가 아니라 **Vercel Serverless Functions** (`FE/api/`) — Vite는 서버 코드를 못 돌리므로, Vercel이 `FE/api/*.js`를 자동으로 서버리스 함수로 인식하는 방식을 쓴다. 로컬 개발 시 프론트(`npm run dev`, 5173포트)와 API는 별도로 동작하며, API 단독 테스트는 `vercel dev` 사용을 권장
- DB: PostgreSQL (Neon, 팀 공용) — 접속/쿼리 로직은 `FE/lib/db/`
- AI 텍스트 생성 + 임베딩: **Gemini API 하나로 통합** (Flash 계열 우선, 정확한 모델명은 Google AI Studio에서 개발 시작 전 확인 — PRD 14.4). 연동 코드는 `FE/lib/ai/`에 추가 예정 (AI/RAG 리드 담당, 아직 미구현)
- Vector DB: ChromaDB (MVP, 아직 미구현)
- 배포: Vercel (GitHub 저장소 연동, `main` 브랜치 push 시 자동 배포). **Vercel 프로젝트의 Root Directory를 `FE`로 설정할 것** (프론트/서버리스 함수가 모두 `FE/` 아래에 있으므로)

## 전공/비전공 구분 없이 기능 구현 가능 (중요)
Claude Code를 쓰면 프로그래밍 경험이 적어도 화면 단위 기능은 직접 구현할 수 있다. PRD 9번의 역할 구분은 "이 사람만 이 일을 한다"가 아니라 "이 사람이 해당 영역을 리드하고 최종 판단한다"는 뜻이다. DB 스키마·AI 파이프라인처럼 구조적 판단이 필요한 영역(PRD 9.1, 9.2)은 전공자가 리드하되, 화면 단위 프론트엔드 기능(PRD 9.3)은 전공/비전공 구분 없이 전원이 Claude Code로 직접 구현한다.

### 비전공자를 위한 Claude Code 활용 가이드
화면 하나를 구현할 때 이 순서로 진행하면 된다.
1. 어떤 화면인지 + PRD 관련 섹션 번호를 함께 알려주며 구현 요청 (예: "PRD.md 6번 UX Flow 참고해서 검색 결과 리스트 화면 만들어줘. shadcn/ui 카드 컴포넌트 쓰고, 로딩·빈·에러 상태도 넣어줘")
2. 만들어지면 `npm run dev` 실행 후 `localhost:3000`에서 직접 눌러보며 확인
3. 이상하게 보이거나 에러가 나면, 에러 메시지나 화면을 그대로 Claude Code에 붙여넣고 "이거 고쳐줘"라고 요청 (코드를 직접 이해하지 못해도 괜찮다)
4. 원하는 대로 동작하면 완료 — 커밋·푸시는 아래 "커밋·푸시 및 충돌 처리" 절차를 따른다

## 환경 설정 (처음 시작할 때 한 번, 팀원 전원 동일하게)
저장소는 이미 만들어져 있다(https://github.com/HoneyPlums/YaHo.git). 실제 프로젝트 루트는 저장소 루트가 아니라 **`FE/` 폴더**다 (모든 명령어를 `FE/` 안에서 실행할 것).

1. Node.js 18 이상 확인: `node -v` (없으면 nodejs.org에서 LTS 설치)
2. 저장소 clone: `git clone https://github.com/HoneyPlums/YaHo.git`
3. `cd YaHo/FE` 로 이동 후 패키지 설치: `npm install`
4. `.env.local` 생성: `FE/.env.local.example`을 복사해서 이름을 `FE/.env.local`로 바꾸고, 실제 값을 채워 넣을 것
   ```
   GEMINI_API_KEY=
   DATABASE_URL=
   ```
5. `.env.local`에 아래 값 채우기 (실제 값은 채팅으로 공유받을 것):
   - `GEMINI_API_KEY` — aistudio.google.com에서 발급, 카드 등록 없이 무료. 텍스트 생성과 임베딩 둘 다 이 키 하나로 처리하며, 팀 전체가 같은 키를 공유한다 (한도도 같이 공유되니 아래 "AI 연동 규칙" 참고)
   - `DATABASE_URL` — 팀 공용 PostgreSQL(Neon) 접속 문자열 (아래 "공용 DB" 참고, 이미 발급되어 채팅으로 공유됨)
6. DB 스키마 적용: `npm run db:migrate` (companies/policies/recommendations_cache 테이블 생성, 이미 한 번 적용되어 있으면 안전하게 재실행 가능)
7. 데이터가 준비되면: `npm run db:seed` (`FE/data/companies.csv`, `FE/data/policies.csv`를 읽어 DB에 upsert. 컬럼 형식은 `FE/data/*.example.csv` 참고)
8. `npm run dev` 실행 후 `localhost:5173` 접속해서 화면이 뜨면 세팅 완료 (`.env.local`의 `DATABASE_URL`은 프론트 번들에 노출되지 않고 `FE/api/`의 서버리스 함수에서만 쓰인다 — Vite에서 `VITE_` 접두사가 없는 환경변수는 클라이언트에 노출되지 않으므로 절대 `VITE_DATABASE_URL`처럼 바꾸지 말 것)

**참고**: 폴더 구조나 설정 파일(package.json 등)은 이미 저장소에 커밋되어 있으므로 `git clone`(또는 이후 `git pull`) 한 번이면 전부 그대로 받아진다. `.gitignore`로 제외되는 `node_modules`(→ `npm install`로 채움)와 `.env.local`(→ 직접 값 채움)만 각자 로컬에서 준비하면 된다.

### 공용 DB (중요)
**팀원마다 로컬에 PostgreSQL을 따로 설치하지 않는다.** 각자 로컬 DB를 쓰면 데이터가 서로 달라져서 협업이 깨진다. Backend/DB 담당(전공자 A)이 Neon PostgreSQL을 이미 만들어 `DATABASE_URL`을 팀 전체에 공유했고, 스키마도 적용 완료했다 — 팀원은 그 값을 각자 `.env.local`에 채우기만 하면 된다. `DATABASE_URL`은 절대 커밋하거나 공개 채널에 올리지 말고, 팀 채팅 DM 등으로만 공유할 것.

### 로컬 ChromaDB 동기화 (중요)
PostgreSQL과 달리 **ChromaDB는 각자 로컬에 따로 존재한다.** 데이터가 준비되면(약 11:00) 그리고 `data/companies.csv`가 `git pull`로 갱신될 때마다, 팀원 각자 `scripts/`의 인덱싱 스크립트를 로컬에서 실행해서 본인의 ChromaDB를 채워야 한다. 이 스크립트를 안 돌리면 다른 사람은 검색이 되는데 본인만 안 되는 상황이 생긴다. 자세한 이유는 PRD 13.4 참고.

## 디렉토리 구조 및 담당 (모두 `FE/` 아래)
- `FE/src/pages`, `FE/src/components` — 화면/컴포넌트 (프론트엔드 기능 구현 담당자, PRD 9.3 참고)
- `FE/api/` — Vercel Serverless Functions, 백엔드 API (Backend/DB). 예: `api/companies/index.js`(검색), `api/companies/[id].js`(상세+정책), `api/policies/index.js`
- `FE/lib/ai/` — Gemini 임베딩·생성, ChromaDB 검색 로직 (AI/RAG, 아직 미구현 — PRD 9.2)
- `FE/lib/db/` — PostgreSQL 쿼리(`queries.js`)·연결(`client.js`)·스키마(`schema.sql`) (Backend/DB)
- `FE/data/` — 실데이터(BEPA 청끌기업). `2026_청끌기업.csv`(기업 120개), `잡아드림_기업정책17개.csv`(청년정책 17개)
- `FE/scripts/` — `migrate.js`(스키마 적용), `seed.js`(위 두 CSV → DB, `npm run db:seed`), 추후 ChromaDB 인덱싱 스크립트도 여기 추가
- `FE/.env.local.example` — 필요한 환경변수 키 이름 목록 (값은 커밋하지 않음)
실제 구조가 달라지면 이 목록을 바로 갱신할 것. 담당 외 디렉토리는 임의로 구조를 바꾸지 말고, 필요하면 채팅으로 먼저 공유할 것.

## 기능 우선순위 (PRD 4번)
- P0 (필수): 기업 검색, AI 맞춤 추천, 추천 이유 생성, 기업 상세 페이지, 청년 지원정책 추천
- P1 (여유 있으면): 신청 링크 연결, 반응형 웹
- P2 (시간 남을 때만): 즐겨찾기, 최근 검색 기록, 기업 비교
- P3 (P0~P2 다 끝났을 때만): 떠다니는 키워드 버블 등 재미 요소
- 시간이 부족하면 P0부터 완성하고, P1 이후는 과감히 보류할 것. P3는 P0가 완전히 끝나기 전까지 절대 먼저 손대지 않을 것

## AI 연동 규칙 (PRD 14번)
- 텍스트 생성(추천 이유)과 임베딩(벡터 검색) 모두 **Gemini API** 하나로 처리 — 별도 임베딩 전용 서비스 쓰지 않음
- 검색 Top-K는 5개 고정
- 유사도 임계값(threshold)은 실데이터로 테스트 전까지 미확정 상태 — 코드에 상수로 빼두고 `// TODO: 실데이터 테스트 후 확정` 주석을 남길 것. 임계값 없이는 "검색 결과 없음" 상태가 아예 발생하지 않으니 반드시 넣을 것
- Gemini 호출은 Top-K 전체를 한 번에 처리하는 배치 방식으로 구현 (기업별로 나눠 호출하지 않기 — 무료 티어 한도 절약에 특히 중요)
- Gemini는 유사도 검색 순서를 재정렬하지 않고 이유만 생성 (범위를 벗어난 임의 확장 자제)
- **Gemini 무료 티어 한도는 API 키가 아니라 프로젝트 전체 기준으로 공유된다.** 팀원 여러 명이 동시에 테스트하면 금방 한도에 걸릴 수 있으니, 테스트 시간을 서로 조율하고 `recommendations_cache`를 적극 활용해 같은 질의를 반복 호출하지 않을 것

## 상태 처리 (PRD 7번)
비동기 데이터를 다루는 화면에는 기본으로 로딩(스켈레톤)/에러/빈 상태 3가지를 구현할 것. 별도 접근성·키보드 네비게이션 작업은 하지 않고 shadcn/ui 기본 동작을 그대로 사용할 것.

## GitHub 협업 규칙

### 브랜치가 충돌을 막아주는 정도 (정확히 알고 가기)
브랜치를 쓰면 **작업하는 동안** 서로 같은 파일을 실시간으로 건드리며 부딪히는 일은 없어진다. 하지만 나중에 `main`에 합칠 때, 두 사람이 **같은 파일의 같은 부분**을 서로 다르게 고쳤다면 브랜치를 썼어도 충돌은 그대로 발생한다. 즉 브랜치는 충돌을 0%로 만들어주지 않고, "충돌이 나더라도 내 작업 중간에 갑자기 나는 게 아니라 병합 시점에만 나게" 만들어준다.

다행히 이 프로젝트는 위 "디렉토리 구조 및 담당" 표대로 각자 다른 파일/폴더를 담당하도록 나눠뒀기 때문에, 실제로 충돌이 날 확률은 낮다. 충돌이 날 가능성이 있는 진짜 공용 파일은 `package.json`(다들 패키지를 설치함), 공용 레이아웃(`app/layout.tsx`), 공용 타입 정의 정도다. 이런 파일을 고칠 땐 미리 채팅으로 알리고, 브랜치를 오래 묵히지 말고 자주 `main`과 동기화할 것.

### 저장소 및 브랜치
- **브랜치 전략**: `main`은 항상 실행 가능한 상태로 유지. 기능 단위로 `feat/기능이름` 같은 짧은 브랜치를 파서 작업하고, 로컬에서 동작 확인 후 바로 `main`에 머지 (해커톤 특성상 PR 리뷰 절차는 생략해도 무방)
- **`main`에 직접 push하지 않기**: 항상 브랜치에서 작업 후 머지할 것. 특히 13:30~14:00 배포·리허설 구간에는 검증 안 된 코드를 `main`에 올리지 않기 — Vercel이 `main` 기준으로 자동 배포되어 데모 직전 화면이 깨질 수 있음
- **커밋 메시지**에 담당 기능을 짧게 명시 (예: `feat: 기업 검색 API 추가`)
- **환경변수**: `.env.local`은 각자 로컬에만 생성하고 절대 커밋하지 않기. 필요한 키 이름은 `.env.local.example`에 커밋하고, 실제 값은 채팅으로 공유
- **새로 작업을 시작하기 전이나 `git pull`/머지로 `main`이 갱신됐을 때, `package.json`이 바뀌었다면 `npm install`을 다시 실행할 것** (누군가 새 패키지를 추가했을 수 있음). `data/companies.csv`가 바뀌었다면 "로컬 ChromaDB 동기화"도 다시 실행할 것

### 커밋·머지·푸시 및 충돌 처리 (Claude Code 공통 동작)
기능 구현을 완료하고 **사용자(작업자 본인)가 확인·승인하면**, Claude Code가 아래 순서를 전부 자동으로 진행한다. 사용자 확인 없이 먼저 진행하지 않는다.

1. 작업 중이던 `feat/기능이름` 브랜치에서 `git add` → `commit`
2. `git checkout main` → `git pull origin main` (최신 `main` 받기)
3. `git merge feat/기능이름` 으로 방금 작업한 브랜치를 `main`에 병합
4. **충돌이 없으면** 그대로 `git push origin main`까지 자동으로 완료한다 (별도 확인 불필요, 이미 3번에서 사용자 승인을 받은 작업이므로)
5. **충돌이 나면** 자동 진행을 멈추고, 아래 "충돌 처리" 절차에 따라 설명하고 사용자의 결정을 기다린 뒤, 결정된 대로 병합을 마무리하고 push한다
6. 병합이 끝난 `feat/기능이름` 브랜치는 로컬에서 삭제해도 좋다고 안내한다 (`git branch -d feat/기능이름`)

임의로 한쪽을 지우거나 `git push --force`를 쓰지 않는다.

**충돌 처리 절차**
1. 충돌난 파일과 충돌 부분(`<<<<<<<` / `=======` / `>>>>>>>`)을 그대로 보여준다
2. 각 쪽이 무엇을 바꾼 것인지(커밋 메시지·diff 기준) 사람이 이해할 수 있게 설명한다
3. 두 변경이 서로 다른 부분을 건드린 것이라 함께 유지해도 되는 경우, 병합안을 제안한다
4. 같은 부분을 다르게 고친 경우라 선택이 필요하면, 무엇이 다른지 짚어주고 사용자에게 어느 쪽으로 할지 물어본 뒤 답을 기다린다
5. 결정이 나면 그 내용대로 충돌을 해소하고, `git add` → `git commit`(머지 커밋) → `git push origin main`까지 마무리한다

남의 담당 파일(위 디렉토리 표 기준)을 수정해야 할 일이 생기면 먼저 채팅으로 알릴 것

## 하지 말아야 할 것
- API 키·DB 접속정보를 코드에 하드코딩하거나 커밋하지 않기 (`.env.local`에만 저장, `.gitignore` 확인)
- P2/P3 기능에 P0보다 먼저 손대지 않기
- Gemini 외에 결제 수단 등록이 필요한 API(예: 사용 중 카드 정보를 요구하는 서비스)를 도입하지 않기 — 대회 규정 위반 위험
- 여러 명이 동시에 Gemini API를 몰아서 테스트하지 않기 (한도 공유)
- 담당 외 디렉토리 구조를 임의로 변경하지 않기
- `main`에 직접 push하지 않기, 검증 안 된 코드를 배포 직전 시간대에 머지하지 않기
- 사용자 확인 없이 push하거나, 충돌을 임의로 해결(강제 push 포함)하지 않기

## 검증 방법
- 자동 테스트 없음 (해커톤 MVP) — 수정 후 핵심 플로우(검색 → 결과 → 상세 → 정책 → 신청)를 직접 클릭해서 확인할 것
- 머지 전 로컬에서 최소한 본인이 만든 기능이 동작하는지 확인할 것
- ChromaDB는 Vercel 서버리스 환경에서 영속성 이슈가 있을 수 있으니, 최종 배포 전이 아니라 개발 초반에 미리 한 번 배포·확인해볼 것 (PRD 12번 참고)
- 배포 전 Vercel 프리뷰에서 반응형(PC/모바일) 확인할 것
- 최종 점검은 PRD 11.2 데모 시나리오 순서대로 한 번 실행해볼 것
